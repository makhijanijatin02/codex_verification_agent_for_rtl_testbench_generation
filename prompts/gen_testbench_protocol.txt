STRICTLY AVOID THESE ASSUMPTIONS UNLESS THE SPEC STATES THEM EXPLICITLY:
- exact synchronization latency across clock domains
- exact number of cycles until data becomes visible in another domain
- exact status-counter values or exact arithmetic of internal counters
- exact equality between internal queue depth and exposed status outputs
- exact returned-credit pulse windows or exact pulse counts based only on an inferred microarchitecture
- bounded wait loops of the form "must happen within N opposite-domain cycles"
- exact truth-table checking for derived 1-bit arithmetic signals such as `credit_available = credit_count - credit_withhold` when the subtraction semantics are not explicitly spelled out
- exact empty-state `pop_valid` / `pop_data` behavior for an empty FIFO unless the spec explicitly defines that case
- exact same-cycle `empty` / `full` / `items` / `slots` values during an empty-state bypass transfer unless the spec explicitly defines those status outputs for that cycle
- exact `items`, `slots`, `items_next`, or `slots_next` equality on every cycle if acceptance semantics around empty/full/bypass edges are not fully specified
- exact equality checks on derived status outputs are lower priority than interface-legality checks; omit them if they risk overspecifying the module
- global equivalence assertions such as "signal X must always equal a reconstructed formula" when that formula depends on ambiguous protocol arithmetic or underspecified same-cycle status behavior

ORACLE STRATEGY:
- For protocol-heavy modules, the oracle should be built from guaranteed interface properties, not speculative internal state.
- For synchronous-reset modules, do not assert reset-state outputs immediately on reset assertion before the first qualifying clock edge unless the specification explicitly defines asynchronous reset behavior.
- Apply reset, advance the required active clock edge, and then check reset-state outputs.
- For synchronous-reset protocol modules, perform the primary reset-state checks with idle handshake inputs unless the specification explicitly defines reset-overlap behavior with active `push_valid`, `pop_ready`, `valid`, or `ready` signals.
- Do not assume that combinational bypass/preview paths are fully suppressed during a reset cycle with active transfer requests unless the specification says so explicitly.
- Do not make strict `*_next` checks during reset unless the specification explicitly defines those predictive outputs under reset.
- If a property is ambiguous, omit the strict check rather than inventing a precise timing or numeric rule.
- Prefer fewer reliable checks over many speculative checks.
- Do not reconstruct internal queue depth, exact credit totals, or exact cross-domain visibility timing unless those semantics are explicitly guaranteed.
- For FIFOs, a conservative oracle may check:
  - reset outputs
  - ready/valid legality
  - same-cycle bypass only in explicitly guaranteed bypass conditions
  - FIFO ordering for clearly accepted buffered transactions
  - no duplication or spontaneous corruption of visible buffered data
  - stability of visible data while backpressured
  - a few exact status checks only in stable, explicitly clocked states rather than at ambiguous same-cycle empty/full edges
  - when a buffered pop occurs, check that `pop_valid` was asserted and `pop_data` matched the old head on the handshake cycle; after the clock edge, expect the next head or other legal post-pop behavior rather than rechecking the old head
- For credit-return modules, a conservative oracle may check:
  - reset blocks `pop_valid` and `push_credit`
  - `push_receiver_in_reset` follows only the guaranteed reset source
  - `pop_data` direct path and `pop_valid` gating
  - `push_credit` is suppressed by reset and stall
  - `push_credit` never asserts when there is clearly no sendable credit under an unambiguous condition
- For `credit_receiver`, prefer a small but reliable checker over an ambitious credit arithmetic model.
- For CDC FIFO / credit modules, a conservative oracle may ignore some count/status outputs entirely if those outputs are not explicitly defined cycle-by-cycle in the specification.

SEQUENTIAL SAFETY RULES:
- Avoid race-prone helper tasks that accidentally consume extra clock edges between consecutive checks
- Keep each task invocation aligned with the intended event being tested
- If waiting for an event, do not hardcode a latency bound unless the specification gives one
- For synchronous resets, never perform the primary reset-value check before the active clock edge that actually applies reset.
